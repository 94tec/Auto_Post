## Quick Start

### Prerequisites

- Node.js **18+**
- Firebase project (Auth + Realtime Database + Firestore enabled)
- Redis Cloud instance ‚ÄĒ [free tier](https://redis.io/try-free/) works fine
- Resend account for transactional email

### Install Both Apps

```bash
git clone https://github.com/your-org/damuchi.git
cd damuchi

# Frontend
cd frontend && npm install

# Backend
cd ../backend && npm install
```

### Seed Your Admin

Before the first run, set your admin directly in Firebase Realtime Database:

```json
users/{your-uid}/basic: {
  "uid":           "your-uid",
  "email":         "admin@yourdomain.com",
  "role":          "admin",
  "status":        "active",
  "emailVerified": true,
  "adminApproved": true,
  "createdAt":     "2024-01-01T00:00:00.000Z"
}

users/{your-uid}/permissions: {
  "read": true, "write": true, "delete": true,
  "manageUsers": true, "accessAdmin": true
}
```

> This seeded admin has no `createdBy` field ‚ÄĒ it bypasses the `mustChangePassword` flow entirely and is never forced to reset.

## How Auth Works

Damuchi uses a **custom token flow** ‚ÄĒ the backend validates your credentials, checks application-level gates, then issues a Firebase custom token for the client SDK to consume. The client never signs in directly with email/password.

```
LoginForm
  ‚Ēā
  ‚ĒĒ‚ĒÄ‚Ėļ POST /api/auth/login  { email, password }
            ‚Ēā
            ‚Ēú‚ĒÄ Firebase REST API signInWithPassword ‚Üí idToken
            ‚Ēú‚ĒÄ Admin SDK verifyIdToken (revocation check)
            ‚Ēú‚ĒÄ Load RTDB profile ‚Üí state machine checks
            ‚Ēú‚ĒÄ Set httpOnly session cookies + CSRF token
            ‚ĒĒ‚ĒÄ createCustomToken(uid)
            ‚Ēā
            ‚Ėľ
      { customToken, user, mustChangePassword? }
            ‚Ēā
            ‚ĒĒ‚ĒÄ‚Ėļ signInWithCustomToken(auth, customToken)
                        ‚Ēā
                        ‚Ėľ
                  onAuthStateChanged fires
                        ‚Ēā
                        ‚Ėľ
                  AuthContext dispatches fetchUserRole(uid)
                        ‚Ēā
                        ‚Ėľ
                  authSlice reads RTDB users/{uid}/basic
                        ‚Ēā
                        ‚Ėľ
                  Redux store populated
                        ‚Ēā
                        ‚Ėľ
                  AuthGuard routes to the right destination
```

**Why not let the client sign in directly?** Firebase Auth has no concept of `SUSPENDED`, `AWAITING_APPROVAL`, or `mustChangePassword`. Those are application-level states. The backend must own that gate ‚ÄĒ and it does.

## Registration ‚ÄĒ 14 Steps, One Atomic Flow

Every step runs in sequence. Any failure after step 8 triggers a **full rollback** ‚ÄĒ Firebase Auth user, RTDB record, Firestore record, and Redis cache entries are all deleted.

```
POST /api/auth/register  { name, email, password }

 1   Field presence             ‚Üí 400 MISSING_FIELDS
 2   Email format (RFC regex)   ‚Üí 400 INVALID_EMAIL
 3   Name length 2‚Äď50           ‚Üí 400 INVALID_NAME_LENGTH
 4   Password strength (5 rules)‚Üí 400 WEAK_PASSWORD
 5   Redis NX lock (30s)        ‚Üí 429 REGISTRATION_IN_PROGRESS
 6   Redis fast dup check O(1)  ‚Üí 409 EMAIL_EXISTS
 7   Firebase Auth dup check    ‚Üí 409 EMAIL_EXISTS / PENDING_VERIFICATION
 8   admin.auth().createUser()  ‚Üź Firebase Auth user created
 9   RTDB multi-path write      ‚Üź role=guest, status=pending
10   Firestore mirror write     ‚Üź parallel with step 9
11   Redis cache warm           ‚Üź email‚Üíuid (7d), uid‚Üíprofile (2h)
12   Release NX lock
13   sendWelcomeEmail()         ‚Üź fire-and-forget
14   AuditLog.record()          ‚Üź fire-and-forget

‚Üí 201 { success, message, nextSteps[], user{} }

ROLLBACK (steps 9‚Äď11 fail):
  admin.auth().deleteUser(uid)
  adminDb.ref(`users/${uid}`).remove()
  firestore.doc(uid).delete()
  rDel(emailToUid), rDel(profile)
```

### Password Rules

Five rules, all checked server-side:

```
‚úď  8+ characters
‚úď  One uppercase   [A-Z]
‚úď  One lowercase   [a-z]
‚úď  One number      [0-9]
‚úď  One symbol      [^A-Za-z0-9]
```

The frontend `RegisterForm` mirrors these with an animated 5-bar strength meter and rule checklist. The backend is the source of truth ‚ÄĒ the UI is just helpful feedback.

## Login ‚ÄĒ A State Machine

The login controller runs five ordered checks before issuing any token. Checks are ordered specifically to prevent information leakage ‚ÄĒ a suspended user cannot discover their verification state by probing.

```
SUSPENDED           ‚Üí 403  ACCOUNT_SUSPENDED     (always checked first)
!emailVerified      ‚Üí 403  EMAIL_NOT_VERIFIED    (non-admin only)
!adminApproved      ‚Üí 403  AWAITING_APPROVAL     (non-admin only)
status = pending    ‚Üí 403  PENDING_ACTIVATION    (edge case guard)
mustChangePassword  ‚Üí 200  + flag in response    (admin + createdBy only)
‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ‚ĒÄ
All pass            ‚Üí 200  { customToken, user }
```

**Admin bypass:** Admins skip the email verification and admin approval gates entirely. An admin account is always fully active ‚ÄĒ it only hits the `mustChangePassword` check if it was created by another admin (the triple guard: `mustChangePassword && role === 'admin' && createdBy`).

## API Client

`src/utils/api.js` is the single HTTP gateway for the frontend. Every request goes through it.

### What It Does Automatically

- Attaches `Authorization: Bearer {firebaseIdToken}` on every request
- Reads `__csrf` cookie ‚Üí sends as `X-CSRF-Token` on all mutating requests
- Silently refreshes the Firebase token on `401 TOKEN_EXPIRED` and retries once
- `authApi.login()` calls `/api/auth/login`, receives `customToken`, and calls `signInWithCustomToken` ‚ÄĒ no manual SDK calls needed

### Usage

```js

await authApi.login({ email, password });
await authApi.forgotPassword(email);
await authApi.resetPassword({ oobCode, newPassword });

// Quotes

await quotesApi.create({ text, author, category });
await quotesApi.update(id, { text });
await quotesApi.delete(id);

// Admin
await adminApi.approveUser(uid);
await adminApi.createAdmin({ email, password, displayName });
await adminApi.grantWrite(uid);
await adminApi.suspendUser(uid);
const { stats } = await adminApi.getStats();
const { queue } = await adminApi.getQueue();

// User account
await userApi.changePassword(newPassword);
await userApi.updateProfile({ displayName });
await userApi.deleteAccount();
```

### Error Handling

The `ApiError` class carries the full backend error shape:

```js

  await authApi.login({ email, password });
```

## Redis Architecture

Redis is not optional infrastructure ‚ÄĒ it is woven into the request lifecycle for registration locks, duplicate detection, profile caching, session data, and cooldowns.

### Key Schema

```
reg:email:{email}         uid string        7 days    email ‚Üí uid lookup
reg:uid:{uid}             JSON profile      2 hours   warm profile cache
reg:lock:{email}          '1'               30 sec    registration idempotency lock
resend:cool:{email}       '1'               2 min     resend verification cooldown
session:{uid}             session JSON      1 hour    session data
vlock:{uid}:{oobSuffix}   '1'               60 sec    email verify idempotency lock
```

### Helpers ‚ÄĒ All Non-Throwing

```js
// authHelpers.js
await rGet(K.profile(uid));                         // null on miss or Redis down
await rSet(K.emailToUid(email), uid, TTL.EMAIL_UID); // silent on error
await rDel(K.profile(uid));                         // safe cache bust
await rSetNX(K.regLock(email), '1', TTL.REG_LOCK);  // null if already locked
```

All helpers are wrapped in try/catch. **A Redis failure never breaks an endpoint** ‚ÄĒ it degrades gracefully to the DB layer.

### Invalidation Pattern

```
User registered    ‚Üí set email‚Üíuid, uid‚Üíprofile
User approved      ‚Üí del uid‚Üíprofile
User suspended     ‚Üí del uid‚Üíprofile
Password changed   ‚Üí del uid‚Üíprofile
Email changed      ‚Üí del old email‚Üíuid, set new email‚Üíuid
Admin creates user ‚Üí set email‚Üíuid, uid‚Üíprofile immediately
```

## The mustChangePassword Contract

When an admin creates another admin via `POST /api/admin/users`, the new account is born with a temporary password and a flag that enforces a password change before anything else happens. This is a full-stack contract ‚ÄĒ backend sets it, frontend enforces it, clearing it requires both sides.

```
Admin A ‚Üí POST /api/admin/users  { email, password, displayName }
              ‚Ēā
              ‚Ėľ
        adminController.createAdmin
          RTDB:       basic/mustChangePassword: true
                      basic/createdBy: AdminA.uid
          Firestore:  mustChangePassword: true
          Redis:      profile cache primed with flag
          Email:      sendAdminWelcomeEmail  (temp password included)
              ‚Ēā
              ‚Ėľ
        Admin B logs in
              ‚Ēā
        loginController triple guard:
          mustChangePassword === true
          && role === 'admin'
          && createdBy !== undefined
              ‚Ēā
              ‚Ėľ
        Response: { success, customToken, mustChangePassword: true }
              ‚Ēā
              ‚Ėľ
        AuthGuard: isAdmin && mustChangePassword
          ‚Üí redirect /auth/change-password  (every navigation, no escape)
              ‚Ēā
              ‚Ėľ
        Admin B sets new password
        POST /api/users/change-password  { newPassword }
              ‚Ēā
              ‚Ėľ
        userController.changePassword
          admin.auth().updateUser(uid, { password })
          admin.auth().revokeRefreshTokens(uid)    ‚Üź all other sessions killed
          Promise.allSettled([
            RTDB:    mustChangePassword: false,
            Firestore: mustChangePassword: false,
            Redis:   rDel(K.profile(uid))           ‚Üź bust cache
          ])
          return { success, mustChangePassword: false }
              ‚Ēā
              ‚Ėľ
        Frontend: dispatch fetchUserRole(uid)
          ‚Üí Redux flag clears
          ‚Üí AuthGuard re-evaluates ‚Üí passes
          ‚Üí navigate('/dashboard')
```

The seeded admin (no `createdBy` field) is **never** subject to this flow. The triple guard fails at the third condition.

## Admin Operations

`adminController.js` manages all admin user operations with Redis cache awareness and full audit trail.

### createAdmin
Creates a new admin account, bypassing the normal registration flow. Sets `mustChangePassword: true` and `createdBy: creatorUid`. Primes Redis immediately. Sends admin welcome email with the temporary password. Records `ADMIN_CREATED` audit event.

### createUser
Creates a pre-approved user ‚ÄĒ skips the entire `guest ‚Üí verify ‚Üí approve` lifecycle. Explicitly sets `mustChangePassword: false` at the RTDB, Firestore, and Redis layer. Regular users are **never** subject to force-change. Records `USER_CREATED_BY_ADMIN` audit event.

### approveUser
Promotes `role=guest` ‚Üí `role=user`, `status=active`. Busts the user's Redis cache. Sends `sendApprovalEmail`. Records both `GUEST_APPROVED` and `GUEST_PROMOTED` events (two events because the state change is meaningful on its own, even if they always happen together).

## Security Model

### Session Cookies

Three cookies, each with a specific role and TTL:

```
__Host-session    httpOnly  secure  sameSite=strict  signed   1 hour
__refresh         httpOnly  secure  sameSite=strict  signed   30 days  path=/auth/refresh
__csrf            NOT httpOnly (client reads it)     signed   1 hour
```

The session cookie appends a 16-character UA fingerprint to the ID token ‚ÄĒ `Buffer.from(userAgent.slice(0,64)).toString('base64').slice(0,16)` ‚ÄĒ to help detect stolen tokens across devices.

### CSRF

All `POST`, `PATCH`, `PUT`, `DELETE` requests require the `X-CSRF-Token` header to match the `__csrf` cookie. The token is regenerated on every login.

### Anti-Enumeration

`forgotPassword` and `resendVerification` always return `200` regardless of whether the email exists in the system. This prevents using the API as an email oracle.

### Session Revocation

After any password change or reset, **all existing sessions are invalidated**:

```js
await admin.auth().revokeRefreshTokens(fbUser.uid);
```
